# Forge AI submission backlog

These items are intentionally not implemented by the submission-focused delivery slice.

| Classification | Issue | Root cause | Impact | Current safe behavior | Recommended future fix |
|---|---|---|---|---|---|
| Backlog | Clickable read-only timeline navigation for completed stages | The timeline is a status display, not a historical document router. | Earlier artifacts are less discoverable. | Current stage-specific documents remain available through their existing controls. | Add immutable stage targets with keyboard-accessible navigation. |
| Backlog | Repository path cannot be reviewed or corrected after requirement approval | Repository selection is bound before analysis and planning. | A mistaken path requires starting a new task. | Forge will not silently rebind approved evidence to another repository. | Add an explicit requirement-revision workflow that invalidates downstream artifacts. |
| Backlog | Dedicated correction-proposal PDF is unavailable | Correction audit is currently included only in the task report. | Sharing one correction proposal separately is inconvenient. | The read-only task PDF retains bounded correction chronology. | Add a proposal-bound exporter with the same redaction and no-side-effect guarantees. |
| Technical debt | Historical full implementation diff may depend on current browser cache | The API persists bounded diff previews, not an unlimited historical rendering artifact. | A full historical diff can be unavailable after navigation or restart. | Truncation remains explicit and persisted review metadata remains truthful. | Persist a bounded content-addressed diff artifact or reconstruct it read-only from retained task worktrees. |
| Backlog | One correction revision maximum | The submission workflow intentionally bounds correction scope and history. | A second failed correction safely stops. | No additional provider dispatch or repository mutation is offered. | Design multi-revision governance and verification lineage. |
| Backlog | Guided recovery UI for ambiguous provider or Git states | Recovery currently prioritizes avoiding duplicate billable or externally mutating actions. | Operators receive a safe stop without guided reconciliation. | Forge preserves evidence and exposes no blind retry. | Add read-only reconciliation diagnostics and explicit operator decisions. |
| Backlog | Automated target validation is unavailable | The submission milestone records only human verification. | Forge cannot independently establish build/test quality. | Every validation command is labelled unexecuted and manual results are user reported. | Add separately approved sandboxed validation execution. |
| Backlog | Delivery recovery is intentionally safe-stop only | Commit, push, or PR ambiguity can make retries unsafe, and this slice has no generalized recovery UI. | Delivery may require manual operator inspection. | Forge creates no second branch or PR, performs no automatic redispatch, and limits same-command reconciliation to exact read-only proof. | Add an auditable operator-guided UI around the bounded commit/remote/PR reconciliation checks. |
| Technical debt | Contradictory legacy implementation runtime message | Some legacy persisted completed results can retain an interruption-era runtime message. | A completed artifact may be paired with “previous attempt was interrupted.” | Persisted artifacts remain authoritative and no mutation occurs during reads. | Add an explicit migration or a provenance-aware projection rule. |
| Backlog | GitHub Enterprise and configurable remote/base support | The first delivery profile is deliberately fixed to GitHub.com, origin, and main. | Other hosts and branch policies are unsupported. | Unsupported remotes and bases are rejected before mutation. | Add an approved administrator configuration model with host allowlists. |
| Backlog | Automatic task-owned worktree cleanup | Retained worktrees preserve review and recovery evidence. | Local disk usage can grow. | Forge does not automatically delete potentially needed evidence. | Add explicit lifecycle policy and recoverable, task-owned cleanup. |
| Backlog | Final UI polish: duplicate diff link, failure Title border, clickable timeline stages | Submission work prioritized workflow safety over adjacent visual cleanup. | Minor duplication and affordance inconsistencies remain. | Existing controls remain keyboard-operable and functional. | Remove the duplicate in-case diff link, add a visible failure-title border, and implement read-only timeline navigation. |
| Backlog | Prevent selecting the Forge AI source repository itself as a delivery target | Repository selection currently validates target safety but does not compare it with Forge AI's own source identity. | An operator could accidentally prepare delivery against the application repository. | Delivery still requires an exact approved task, isolated worktree, explicit proposal approval, and clean-checkout protection. | Bind a deployment-safe Forge source identity and reject an equal target before repository analysis or delivery preparation. |
| Technical debt | Reconstruct and validate execute-command bindings during every task read | Execute bindings are checked on replay, while ordinary reads validate the normalized attempt/proposal records rather than reconstructing every execute semantic fingerprint. | Direct database corruption of an execute binding may remain latent until replay. | Proposal, approval, attempt ownership, JSON/relational columns, and current pointers fail closed during reads. | Reconstruct each execute command fingerprint from persisted task state during ordinary reads and cross-check the binding ledger. |
| Technical debt | Verify delivery worktree branch and HEAD immediately before staging | Exact worktree content and status are checked before staging, while branch/HEAD identity is ultimately proven through the created commit parent. | An unexpected pre-staging branch/ref change is detected later than ideal, after local mutation may have begun. | Exact paths/content must match and the commit parent must equal the approved base; ambiguity enters recovery-required. | Add an explicit read-only branch and HEAD equality check immediately before `git add`. |
| Technical debt | Independently bind frontend PR URLs to the proposal repository | The strict frontend decoder validates canonical GitHub PR URL shape but relies on backend relationship validation for owner/repository equality. | A malformed server response with another canonical GitHub repository URL is not independently rejected by the browser. | Backend creation and persistence require the canonical URL reconstructed from the approved owner/repository. | Compare decoded URL owner/repository and PR number directly with the proposal and successful attempt in frontend relationship validation. |
